# コメント機能の仕様書

このドキュメントでは、なでしこ3貯蔵庫（nako3storage）に追加された作品へのコメント機能の仕様について記述します。

---

## 1. 概要
作品ページ下部において、ユーザーが簡単なコメントや返信を投稿・共有できるスレッド形式のコミュニケーション機能です。荒らし・いたずら対策を幾重にも施したセキュアな設計になっています。

---

## 2. データベース仕様
SQLite のメインDB（`data/n3s_main.sqlite`）に保存されます。

### 拡張されたテーブル・新規テーブル

#### 1. `comments` テーブル (コメント本体)
- `comment_id`: INTEGER PRIMARY KEY
- `user_id`: INTEGER DEFAULT 0 (投稿したユーザーのID)
- `app_id`: INTEGER DEFAULT -1 (対象の作品ID)
- `parent_id`: INTEGER DEFAULT 0 (親コメントのID。0の場合は新規スレッド)
- `name`: TEXT (投稿者名)
- `body`: TEXT (本文)
- `ip`: TEXT (IPアドレス)
- `status`: TEXT DEFAULT 'pending' (状態: `pending`＝審査中, `approved`＝公開, `ng`＝非公開)
- `fav`: INTEGER DEFAULT 0 (いいね数)
- `ctime` / `mtime`: INTEGER (作成・更新UNIXタイム)

#### 2. `comment_likes` テーブル (重複いいね防止用)
- `comment_like_id`: INTEGER PRIMARY KEY
- `user_id`: INTEGER (いいねしたユーザーのID)
- `comment_id`: INTEGER (対象コメントID)
- `ctime`: INTEGER (作成タイム)
- `UNIQUE(user_id, comment_id)` による二重いいねの防止

#### 3. `comment_audit_cache` テーブル (Gemini API 料金節約用キャッシュ)
- `body_hash`: TEXT PRIMARY KEY (コメント本文の SHA-256 ハッシュ)
- `result`: TEXT DEFAULT '' (判定結果: `approved` / `ng`)
- `reason`: TEXT DEFAULT '' (判定理由)
- `ctime`: INTEGER (キャッシュ作成タイム)

### 自動マイグレーション
`app/n3s_lib.inc.php` 内の `n3s_db_init()` 時に `n3s_db_migrate_comments()` が実行され、既存の古いDBファイルに対しても上記テーブルの作成やカラム追加が自動で行われます。

## 3. セキュリティといたずら対策

### 1. ログイン制限
コメントの投稿、いいね、および削除機能の利用は**ログイン中ユーザーのみ**に制限されます。未ログイン時はログインを促すリンクが表示されます。

### 2. NGワードフィルタ
`n3s_config.ini.php` などで指定された `$n3s_config['ng_words']` （配列）に含まれる単語がコメント本文内にある場合、保存を拒否してエラーメッセージを返します。

### 3. 生成AI（Gemini API）による自動審査バッチ
投稿された直後のコメントはすべて `status = 'pending'`（審査中）として登録され、即座には一般公開されません。
1時間に1回程度、以下の審査バッチを定期実行して公開を判定します。

- **バッチスクリプト**: `scripts/comment_audit.php`
- **実行コマンド**: `just comment-audit`（内部で `php scripts/comment_audit.php` を実行）
- **動作仕様**:
  - `status = 'pending'` のコメントを取得し、Gemini API を使って誹謗中傷やスパムなどを審査します。
  - **APIキー未設定時のフォールバック**: 設定ファイルで `gemini_api_key` が設定されていない場合、または `comment_audit_auto_approve` が有効な場合は、AI審査をパスして自動で `approved` に自動承認（公開）されます。
  - **一時的エラー発生時の挙動（判定結果の維持）**: APIサーバーからモデル終了等のエラーレスポンス（code: 404など）や通信エラーが返ってきた場合は、そのコメントを「不承認」にするのではなく、ステータスを `pending` のまま維持して処理を保留します。
  - **管理者へのエラーメール通知**: 審査対象のコメントが1件以上ある中で、判定処理を行ったコメントの全てがエラー（404等）になった場合は、管理者のメールアドレス（`admin_email`）宛てに `Gemini APIの状態を確認するようにして。(スクリプトパス)(サーバー名)` という警告メールを送信します。

### 4. API料金節約用キャッシュ
自動審査バッチが Gemini API を呼び出す際、同一のコメント本文が再度投稿された場合（スパムの繰り返しなど）にAPI料金を浪費しないよう、審査結果を `comment_audit_cache` にキャッシュします。
2回目以降の同一本文の判定は、APIを呼び出すことなくキャッシュ結果が即時に適用されます。

### 5. 審査中コメントのマスク処理
審査中（`pending`）のコメントもスレッド構造維持のために一覧には返されますが、**APIサーバー側で本文を `(現在内容を審査中…)` に上書きして配信**します。これにより、審査前の不適切な内容やスパムテキストがブラウザ側に漏出することはありません。

### 6. 不承認（ng）コメントの保持とマスク処理
Gemini APIによる自動審査で不承認（`ng`）となったコメントは、将来的なユーザーからのクレームや事実確認に対応できるよう、**DB上には投稿時の生データをそのまま保持**します。
しかし、APIサーバーからクライアントへ返却される際には、本文が強制的に `(AIによりコメントが却下されました)` に上書きして配信されます。

### 7. コメント削除権限とカスケード削除
- **権限**: 投稿者本人（`user_id`が一致）または管理者（`n3s_is_admin()`）のみ削除可能です。
- **削除時の挙動**: 親コメントが削除された場合、スレッド整合性を保つため、その配下の子コメント（返信）およびそれぞれのいいね情報もまとめて物理削除（カスケード削除）されます。

## 4. UI/UX 仕様

### 1. アコーディオン表示と状態保持
- コメントセクションは、初期表示ではアコーディオン形式で閉じられており、「コメントが＊件あります」とだけ表示されます。
- ヘッダーをクリックすると開き、コメント一覧と投稿フォームが表示されます。
- 一度開いた開閉状態は `localStorage` に保持（キー: `n3s_comments_open_{app_id}`）され、リロード後も状態が維持されます。

### 2. スレッドツリー表示とソート
- ユーザーのコメントは2階層のスレッドツリー形式（親コメントと、1段インデントされた子コメントへの返信）で表示されます。
- ソート順は「**スレッド全体のいいね合計（親＋子の合算）が多い順**」に並びます。いいね数が同じ場合は新着順になります。
- 審査中（`pending`）のコメントは名前の横に黄色文字で `(現在内容を審査中…)`、不承認（`ng`）のコメントは名前の横に赤色文字で `(AIにより却下されました)` ラベルが表示されます。
- `pending` または `ng` ステータスのコメントは、本文の文字色がグレー（`#888`）の斜体で表示され、いいねボタンや返信フォームが無効化（非表示）されます。

### 3. 返信時の自動フォーカス
- `[返信]` ボタンを押すと対象の返信フォームが展開され、直後に入力テキストエリア (`textarea`) に自動的にフォーカスが当たります。

### 4. インラインメッセージ通知
- コメント送信や返信の完了時に、ブラウザ標準の `alert()` ダイアログは表示されません。
- 代わりに送信フォーム直上に用意された青色のメッセージ領域に、「AIが審査した後に公開されます。しばらくお待ちください。」という内容がインラインで表示され、15秒後に自動的に消去されます。

## 5. コメント API（`api.php?action=comment`）
すべてのリクエストはパラメータ `mode` によって処理が分岐します。

### 一覧取得 (`mode=list`)
- **メソッド**: `GET`
- **パラメータ**: `app_id` (作品ID)
- **返却値**: コメントツリーの配列
  - 審査中 (`status = 'pending'`) のコメントは `body` が `(現在内容を審査中…)` にマスクされます。
  - 不承認 (`status = 'ng'`) のコメントは `body` が `(AIによりコメントが却下されました)` にマスクされます。
  - レスポンス内の各コメントオブジェクトには `can_delete` (boolean) が含まれ、現在のログインユーザーがそのコメントを削除できるかどうかが一目で判定できます。

### コメント投稿/返信 (`mode=add`)
- **メソッド**: `POST`
- **パラメータ**: `app_id`, `parent_id` (0なら新規、親IDなら返信), `body` (本文), `edit_token` (CSRF用)
- **制限**: ログイン必須。

### いいねトグル (`mode=fav`)
- **メソッド**: `POST`
- **パラメータ**: `comment_id`, `edit_token` (CSRF用)
- **制限**: ログイン必須。審査中（`pending`）または不承認（`ng`）のコメントにはいいねできません。

### コメント削除 (`mode=delete`)
- **メソッド**: `POST`
- **パラメータ**: `comment_id`, `edit_token` (CSRF用)
- **制限**: ログイン必須。投稿者本人または管理者のみ。

## 6. テストと検証
`tests/Feature/CommentTest.php` にて、以下のケースを網羅する自動テストが実装されています。

```bash
# テストの実行
just test
```

### 主なテスト内容
1. 未ログイン時の投稿・いいね拒否の検証
2. ログイン時のコメント投稿 (`status=pending` で保存されることの検証)
3. NGワードを含む投稿の拒否
4. 重複いいね防止とトグルの動作検証
5. 親子コメントのスレッドソート（トータルいいね合計数順）の検証
6. 自動審査バッチのシミュレーション（auto_approve および APIキー未指定時の自動承認パスの検証）
7. 審査中（`pending`）コメント一覧取得時の本文マスク処理の検証
8. 不承認（`ng`）コメント一覧取得時の本文マスク（`(AIによりコメントが却下されました)`）およびDB生内容保持の検証
9. 自動審査バッチの一時的エラー（404等）発生時のpending状態保留および管理者へのメール通知アラートの検証
10. コメント削除権限（本人・他者の拒否・管理者特権）およびカスケード削除の検証
11. 同一本文のコメントに対するバッチ審査結果のキャッシュルックアップおよび無用なAPI呼び出し防止の検証
