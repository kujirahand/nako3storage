# コメント機能の仕様書

このドキュメントでは、なでしこ3貯蔵庫（nako3storage）の作品コメント機能に関する最新の仕様について記述します。

---

## 1. 概要
作品ページ下部において、ユーザーが簡単なコメントや返信を投稿・共有できるスレッド形式のコミュニケーション機能です。
荒らし・いたずら・スパム対策とセキュリティ対策を施し、かつ高いパフォーマンスと滑らかな UI/UX を両立させた設計になっています。

---

## 2. データベース仕様
コメント関連データは、メインDB（`data/n3s_main.sqlite`）に保存されます。

### テーブル定義

#### 1. `comments` テーブル (コメント本体)
- `comment_id`: INTEGER PRIMARY KEY
- `user_id`: INTEGER DEFAULT 0 (投稿したユーザーのID)
- `app_id`: INTEGER DEFAULT -1 (対象の作品ID)
- `parent_id`: INTEGER DEFAULT 0 (親コメントのID。0の場合は新規スレッド)
- `name`: TEXT (投稿者名)
- `body`: TEXT (本文)
- `ip`: TEXT (IPアドレス)
- `status`: TEXT DEFAULT 'pending' (状態: `pending`＝審査中, `approved`＝公開, `ng`＝不承認)
- `fav`: INTEGER DEFAULT 0 (いいね数)
- `ctime` / `mtime`: INTEGER (作成・更新UNIXタイム)

#### 2. `comment_likes` テーブル (重複いいね防止用)
- `comment_like_id`: INTEGER PRIMARY KEY
- `user_id`: INTEGER (いいねしたユーザーのID)
- `comment_id`: INTEGER (対象コメントID)
- `ctime`: INTEGER (作成タイム)
- `UNIQUE(user_id, comment_id)` 制約による二重いいねの防止

#### 3. `comment_audit_cache` テーブル (Gemini API 料金節約用キャッシュ)
- `body_hash`: TEXT PRIMARY KEY (コメント本文の SHA-256 ハッシュ値)
- `result`: TEXT DEFAULT '' (判定結果: `approved` / `ng`)
- `reason`: TEXT DEFAULT '' (判定理由)
- `ctime`: INTEGER (キャッシュ作成タイム)

### 自動マイグレーション
`app/n3s_lib.inc.php` 内の `n3s_db_init()` 実行時に自動的に `n3s_db_migrate_comments()` がトリガーされ、稼働中のDBに対しても上記テーブルの作成やカラム追加が自動で行われます。

## 3. セキュリティといたずら対策

### 1. ログイン制限とNGワードフィルタ
- コメントの投稿、いいね、および削除機能の利用は**ログイン中ユーザーのみ**に制限されます。未ログイン時はログインを促すリンクが表示されます。
- `$n3s_config['ng_words']` （配列）に含まれる単語がコメント本文内にある場合、保存を拒否してエラーメッセージを返します。

### 2. 非公開・限定公開作品のコメントアクセス制限
- コメント API アクション（一覧取得、投稿、いいね、削除）のすべてにおいて、対象の作品が非公開 (`is_private = 1`) または限定公開 (`is_private = 2`) であるかを検証します。
- `n3s_private_access_allowed()` を用いた閲覧権限チェックを行い、権限のない第三者からのアクセスに対しては `{"result":false,"msg":"この作品の閲覧権限がありません。"}` と JSON 形式で拒否します。これにより、URL 直叩きによるコメントの漏洩を完全に防ぎます。

### 3. 生成AI（Gemini API）による自動審査バッチ
投稿された直後のコメントはすべて `status = 'pending'`（審査中）として登録され、即座には一般公開されません。
定期実行（cron等）により以下の審査バッチを起動して公開可否を判定します。

- **バッチスクリプト**: `scripts/comment_audit.php`
- **実行コマンド**: `just comment-audit`（内部で `php scripts/comment_audit.php` を実行）
- **審査仕様**:
  - `status = 'pending'` のコメントを取得し、Gemini API を使って誹謗中傷やスパムなどを審査します。
  - **最新の安価なAIモデル**: 高速・軽量でコストパフォーマンスに優れた最新モデル **`gemini-3.1-flash-lite`** を使用して判定を行います。
  - **タイムアウト設定**: API 呼び出しの curl 接続に、**接続タイムアウト（10秒）** と **実行全体タイムアウト（30秒）** を設定し、API サーバー無応答時に cron 実行がハングアップするのを防ぎます。
  - **バージョン互換解放**: PHP 8.0 未満の古い環境でのみ明示的に `curl_close()` を呼ぶように制御し、PHP 8.5 以降での Deprecated 警告を回避しつつ、旧 PHP バージョンでも確実に curl リソースを解放します。
  - **一時的エラー発生時の保留**: APIサーバーからモデル終了等のエラーレスポンス（code: 404など）や通信エラーが返ってきた場合は、そのコメントを「不承認」にするのではなく、ステータスを `pending` のまま維持して処理を保留します。
  - **管理者へのエラーメール通知**: 審査対象のコメントがある中で、バッチ実行した全ての件数がエラー（404等）になった場合は、管理者の `admin_email` 宛てに `Gemini APIの状態を確認するようにして。(スクリプトパス)(サーバー名)` という警告メールを送信します。
  - **APIキー未設定時のフォールバック**: 設定ファイルで `gemini-api-key` が設定されていない場合、または `comment_audit_auto_approve` が有効な場合は、AI審査をパスして自動で `approved` に自動承認（公開）されます。

### 4. API料金節約用キャッシュ
自動審査バッチが Gemini API を呼び出す際、同一のコメント本文が再度投稿された場合にAPI料金を浪費しないよう、審査結果を `comment_audit_cache` にキャッシュします。
2回目以降の同一本文の判定は、APIを呼び出すことなくキャッシュ結果が即時に適用されます（APIキー未設定時の自動承認時はキャッシュされません）。

### 5. 審査中・不承認コメントのマスク処理
- **審査中 (`pending`)**: スレッド構造維持のために一覧には返されますが、APIサーバー側で本文を `(現在内容を審査中…)` に上書きして配信します。
- **不承認 (`ng`)**: クレーム対応や事実確認に対応できるよう、**DB上には投稿時の生データをそのまま保持**します。しかし、APIサーバーからクライアントへ返却される際には、本文が強制的に `(AIによりコメントが却下されました)` に上書きして配信されます。
- 名前の横に重複してステータスを表示することは行わず、本文のマスク処理のみに表示を統一しています。

## 4. パフォーマンス最適化（N+1問題の解決）
コメント数が多くなっても API の応答速度が低下しないように、データベースクエリを定数回に削減しています。

- **いいね情報の一括取得**: 各コメントごとに `comment_likes` テーブルを SELECT していたのを廃止し、取得対象のすべての `comment_id` を収集したうえで、`comment_id IN (...)` 句を用いて「ログイン中ユーザーが過去にいいねしたコメント ID」を一括で1回のみ取得するようにしました。
- **子コメントの一括取得**: 親コメントのループ内で個別に子コメントを `SELECT` していた処理を廃止し、親IDのリストを用いて `parent_id IN (...)` 句により、1回のクエリですべての子コメントをまとめて取得し、メモリ上で親子関係をマッピングします。
- これにより、表示時のデータベースクエリはコメント数に関わらず「親コメント」「子コメント」「いいね情報」の**最大3回**（ログインなし時は2回）に集約されています。

## 5. UI/UX 仕様

### 1. アコーディオン表示と状態保持
- コメントセクションは初期表示ではアコーディオン形式で閉じられており、「コメントが＊件あります」とだけ表示されます。
- 開閉状態は `localStorage` に保持（キー: `n3s_comments_open_{app_id}`）され、リロード後も状態が維持されます。
- 「作品公開情報」ブロックもアコーディオン形式になっており、デフォルトでクローズした状態で表示されます。

### 2. スレッドツリー表示とソート
- コメントは2階層のスレッドツリー形式（親コメントと、1段インデントされた子コメントへの返信）で表示されます。
- ログインユーザーが投稿したコメントは、投稿者名をそのユーザーの作品一覧（`index.php?action=list&user_id=<user_id>`）へのリンクとして表示します。匿名コメントの投稿者名はリンクしません。
- ソート順は「**スレッド全体のいいね合計（親＋子の合算）が多い順**」に並びます。いいね数が同じ場合は新着順になります。
- 審査中（`pending`）または不承認（`ng`）ステータスのコメントは、本文を斜体かつ文字色グレー (`#888`) で表示し、いいねスター（`⭐`）アイコンおよび返信フォームを非表示（無効化）にします。

### 3. 返信時の自動フォーカス
- `[返信]` ボタンを押すと対象の返信フォームが展開され、直後に入力テキストエリア (`textarea`) に自動的にフォーカス (`focus()`) が当たります。

### 4. インラインメッセージ通知
- コメント送信や返信の完了時に、ブラウザ標準の `alert()` ダイアログは表示されません。
- 代わりに送信フォーム直上に用意された青色のメッセージ領域に、「AIが審査した後に公開されます。しばらくお待ちください。」という内容がインラインで表示され、15秒後に自動的に消去されます。

### 5. リソースの外部 JS ファイル化
- コメント関連の JavaScript コードはすべて `app/resource/nako3storage_comment.js` に分離され、キャッシュ防止のためのタイムスタンプ付きの URL で読み込まれます。
- `show.html` 側では、外部 JS が適切に動作するように `window.app_id` や `window.n3s_editkey` などの動的変数の初期化のみを行います。

### 6. コメント削除とカスケード削除
- 投稿者本人（`user_id`が一致）または管理者のみ削除可能です。
- 親コメントが削除された場合、スレッド整合性を保つため、その配下の子コメント（返信）およびそれぞれのいいね情報もまとめて物理削除（カスケード削除）されます。

## 6. コメント API（`api.php?action=comment`）
すべてのリクエストはパラメータ `mode` によって処理が分岐します。

### 一覧取得 (`mode=list`)
- **メソッド**: `GET`
- **パラメータ**: `app_id` (作品ID), `editkey` (閲覧キー、省略可)
- **返却値**: コメントツリーの JSON 配列。閲覧権限がない場合はエラーを返します。

### コメント投稿/返信 (`mode=add`)
- **メソッド**: `POST`
- **パラメータ**: `app_id`, `parent_id`, `body`, `editkey`, `edit_token`
- **制限**: ログイン必須。NGワード判定、閲覧権限チェックを通過する必要があります。

### いいねトグル (`mode=fav`)
- **メソッド**: `POST`
- **パラメータ**: `comment_id`, `editkey`, `edit_token`
- **制限**: ログイン必須。

### コメント削除 (`mode=delete`)
- **メソッド**: `POST`
- **パラメータ**: `comment_id`, `editkey`, `edit_token`
- **制限**: ログイン必須。本人または管理者のみ。
